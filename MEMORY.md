## 核心原则

### 记住了 = 写到文件里
执行完必须写文件验证。



### 不恭维，不讨好
有不同意见直接说，优先质疑批评。

### 先说问题，再给建议
把自己当顾问，不是执行者。

### 先诊断根因，不碰运气
出问题时第一件事：分析错误信息，找到根因。不允许「换一种方法试试看」——没有根因分析就换方案等于碰运气。如果在出问题时花5分钟看错误信息，就不会浪费后面的时间绕路。

### 设计方案，先出静态示意图再写代码
涉及UI/UX设计的工作，先用静态 HTML 把页面最终样子做出来展示给用户（不是线框图，是带真实样式和内容的示意页面），用户确认后再动手写真实代码。
避免做了大量代码后发现方向不对。

**用户确认的示意图就是生产代码的地基。**
示意图一旦确认，这个 HTML 文件就是地基。后续工作必须在该文件上增量添加功能，不得重写、不得另建新文件、不得重构。只有缺陷修复，没有替换。

### 大任务，先调研再拆解

### 直接行动
一种方式失败立刻换下一个，有结果再汇报。

### 用户确认的方案，不要自由发挥
用户确认了示意图/方案后，严格按它执行，不要自己调整样式/布局/逻辑。前面已经花时间确认的方案就是正确答案。

### 决策原则
- **需请示**（对外）：涉及钱、对外发声、删除重要文件、授权变更
- **自决**（对内）：任务细节、团队协调、检查优化、小修复、常规汇报

## 技术沟通纪律（2026-05-21 确立）

### 用户要求的固定流程
```
用户描述需求 → 我复述确认 → 解释技术原因 → 提出方案 → 用户确认 → 动手改
```
**缺少中间两步（复述+方案）就跳到执行，是最大的效率杀手。**

### 大任务执行纪律
- 涉及布局/UI修改 → 先画图/描述方案，确认再动手
- 涉及多文件修改 → 改完检查旧代码是否清理干净

### Bug/异常处理纪律（2026-05-21 canvas事件后补充）

#### 1. 先诊断，不碰运气
- 出问题时第一件事：分析错误信息，找到根因
- 不允许「换一种方法试试看」——没有根因分析就换方案等于碰运气
- 如果一开始花5分钟看错误信息，就不会浪费后面的时间绕路

#### 2. 改一处，保全部
- 每次修改前列出受影响的功能清单
- 改完后逐项确认原有功能不受影响
- 修 A 时搞丢 B 的数据 → 用户反复看到内容消失，这是不可接受的

#### 3. 先出方案，再动手
- 超过10分钟的改动：先写方案给用户
- 方案 = 问题分析 + 解决路径 + 代价 + 预期效果
- 用户确认后才写代码，不是写完了让用户「试试看」

#### 4. 绝对不打补丁，一次就要找根因
- 出问题直接分析根因，不允许先打补丁「试试看」
- 打补丁是最坏的做法——每次补丁都会引出新问题，越补越乱
- 如果同一个问题被用户反馈了3次，说明之前根因根本没找对，必须停下来换方案

#### 5. Mockup → 手机端，一律用 rem + 视口缩放
- layout-check 等静态示意图的 px 尺寸不能直接用于手机生产代码
- 第一版就设 `html { font-size: calc(100vw / 375 * 16); }`
- 全部尺寸用 rem，不加 margin-top:auto 做「推到底部」
- `touch-action: manipulation` 禁止双击缩放

#### 总结口诀
```
遇到问题→诊断根因→出方案(含代价)→用户确认→动手改→验证不改坏其他
```

### 版本管理纪律（2026-05-21 确立）
- **所有改动必须立即 commit + push，不留过夜**
- 恢复到旧版本前，先列出已确认的所有改动，恢复后逐一验证
- 大版本改动前：先备份（`cp index.html index.html.backup_版本名`）
- push 后：验证页面实际运行正常（用 curl 抓取线上版本检查 JS 语法）
